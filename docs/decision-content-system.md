# FiveSeven AI 工具库 / 模型库 — 决策型内容系统升级方案

---

## 1. 工具库 / 模型库的正确角色

### 它们在 FiveSeven AI 中的角色

工具库和模型库不是产品目录，不是百科全书，也不是广告橱窗。

在 FiveSeven AI 的内容体系中，它们的角色是 **决策入口**——用户在这里完成三件事：

1. **判断**：这个工具/模型适不适合我的岗位和场景
2. **选择**：同类工具/模型中我该选哪个
3. **行动**：决定后下一步去哪里学习使用它

工具页和模型页是学习路径的「十字路口」。用户不会在这里停留很久，但这里的信息质量直接决定他后续的路径是否正确。

### 为什么「介绍型页面」价值低

看一下当前工具卡片（以 ChatGPT 为例）展示的全部信息：

```
名称：ChatGPT
描述：一段 50 字左右的概述
定价：免费版可用 / Plus $20/月 / Pro $200/月
擅长解决：文案撰写、数据分析、方案生成、信息整理、翻译润色、深度推理
→ 访问官网 / 查看教程
```

问题出在哪里：

| 用户真正想知道的 | 当前页面给了什么 |
|----------------|----------------|
| 「跟 Claude 比该选哪个？」 | 没有对比 |
| 「免费版够不够用？」 | 只列了价格，没给判断 |
| 「我是运营，用它做什么最有效？」 | suitableJobs 写了 "all"，等于没说 |
| 「上手难吗？需要多久能产出？」 | difficulty 写了 "easy"，但没有上下文 |
| 「有什么做不好的？」 | 完全没有 |

**一个没有判断、没有对比、没有局限性说明的工具页，跟官网首页没有区别。** 用户不需要 FiveSeven AI 替他打开官网。

### 什么样的页面才具备「决策价值」

决策价值 = 用户读完后，能做出一个原来做不出的选择。

具体表现为用户读完能回答这 5 个问题：

1. **我该不该用这个工具？**（适用判断）
2. **它比同类工具好在哪、差在哪？**（对比判断）
3. **花钱值不值？免费版够不够？**（成本判断）
4. **从哪里开始学？**（行动路径）
5. **它做不了什么？遇到这种情况该换什么？**（边界判断）

如果页面不能回答这 5 个问题中的至少 3 个，它就只是介绍，不是决策。

### 各模块写作要求与常见误区

#### verdict（一句话结论）

**写作要求**：40-80 字。必须包含「适合什么」和「不适合什么 / 局限在哪」。用户读完这一句就能做出初步判断。

**常见误区**：
- 写成广告语：「最强大的 AI 工具」→ 没有信息量
- 过于中性：「各有优劣」→ 等于没说
- 只写优点不写边界：「适合所有人」→ 用户不信

**正确示范**：「大多数人的第一个 AI 工具。如果你只能选一个，选它不会错——但处理超长文档或深度分析时，Claude 更强。」

#### bestFor / notFor（适合谁 / 不适合谁）

**写作要求**：各 1 句话，不超过 30 字。用「需要 XX 的人」句式。不要用岗位名，用场景描述。

**常见误区**：
- bestFor 写「all」→ 必须写具体场景
- notFor 留空 → 每个工具必须有不适合的场景
- 用技术术语 → 用户听不懂「token 窗口不够大」，要说「处理超过 5 万字的文档时会丢信息」

#### strengths / limitations（优势 / 局限）

**写作要求**：各 3-5 条。每条不超过 25 字。必须是具体能力，不是感受。

**常见误区**：
- 优势写「性能强大」→ 要写「SWE-bench 80.8%，编码能力业界第一」
- 局限写「有时候不太准」→ 要写「处理 10 万字以上文档时后半段信息利用率下降」
- 优势 5 条局限 1 条 → 比例失衡，用户不信任。至少 3:2

#### pricingAdvice（定价建议）

**写作要求**：不只列价格，必须给出判断——「大多数人用 XX 套餐就够了」。

**常见误区**：
- 只列套餐名和价格 → 这是官网干的事
- 不敢给建议 → 「根据需求选择」等于没说
- 漏掉免费版的真实体验 → 用户最关心免费能干什么

#### vsAlternatives（和替代方案的对比）

**写作要求**：最多对比 2-3 个最常被比较的工具。每组对比必须有结论：「XX 场景选这个，YY 场景选那个」。

**常见误区**：
- 罗列两个工具各自的功能列表 → 用户自己没法从功能列表得出结论
- 「各有优劣，看个人需求」→ 一票否决，必须给出场景化建议
- 对比维度太多 → 最多 3 个维度，聚焦用户最关心的差异

#### quickStart（快速上手）

**写作要求**：一行文字，不超过 30 字。格式：「动作 → 动作 → 动作 → 结果」。

**常见误区**：
- 写了一整段教学 → 这是教程的事，不是工具页的事
- 步骤超过 4 步 → 快速上手 ≠ 完整教学

#### relatedTutorials / relatedPrompts / relatedPaths（推荐搭配阅读）

**写作要求**：教程最多 3 个，Prompt 最多 5 个，路径最多 2 个。每个必须与该工具直接相关。

**常见误区**：
- 推荐了 10 个「相关内容」→ 太多等于没推荐
- 推荐的教程用的不是这个工具 → 必须是直接使用该工具的教程
- 没有推荐 Prompt → 工具页到 Prompt 库是最短的行动路径

### 各模块写作要求

| 模块 | 写作要求 | 常见误区 |
|------|---------|---------|
| positioning（定位） | 一句话说清「什么场景下选它」+ 一句话说清「它的代价是什么」 | 只写「最强模型」不写代价 |
| idealScenarios（适合场景） | 3-5 个，每个用具体任务描述，不用抽象能力词 | 「文本生成」→ 应写「完整策划方案撰写（活动方案、PRD）」 |
| weakScenarios（不适合场景） | 2-3 个，说明为什么不适合，推荐替代方案 | 留空或只写 1 条 |
| coreAdvantages（核心优势） | 必须有数据支撑（基准分数、排名、具体指标） | 「推理能力强」→ 应写「SWE-bench 80.8% 编码第一」 |
| coreWeaknesses（短板） | 必须具体，并说明影响是什么 | 「速度一般」→ 应写「速度慢于 Sonnet——不适合需要即时反馈的场景」 |
| vsModels（模型对比） | 每组必须有一句话结论 + 具体推荐场景 | 罗列参数差异但不下结论 |
| costPerformance（性价比） | 必须给出「什么时候值得花这个钱」的判断 | 只列价格不给建议 |
| recommendedFor（推荐人群） | 按岗位标注，每个岗位说明具体原因 | 写「适合所有人」 |

## 6. 站内联动机制

### 联动设计原则

每个工具页/模型页不是终点，而是路径的中间站。用户在这里做完决策后，必须有明确的下一步。

### 联动关系图

```
                    ┌──────────┐
                    │ 学习路径  │
                    │ (入口)   │
                    └────┬─────┘
                         │ 路径中提到某个工具
                         ▼
                    ┌──────────┐
           ┌───────│  工具页   │───────┐
           │       │ (决策)   │       │
           │       └────┬─────┘       │
           │            │             │
           ▼            ▼             ▼
      ┌────────┐  ┌──────────┐  ┌──────────┐
      │ 模型页  │  │ 实战指南  │  │ Prompt 库 │
      │(深入选型)│  │(学会用它) │  │(立刻用起来)│
      └────────┘  └──────────┘  └──────────┘
```

### 具体联动规则

#### 工具页 → 实战指南

```
规则：工具页底部「推荐搭配阅读」区域最多展示 3 篇教程。
筛选逻辑：tutorialId 直连教程排第一，然后按 targetJobs 匹配展示。
渲染方式：教程卡片（标题 + 难度 + 时长 + solves 一句话）。
文案引导：「学会用 ChatGPT → 从这些教程开始」
```

#### 工具页 → Prompt 库

```
规则：工具页底部展示 3-5 个推荐 Prompt。
筛选逻辑：按 recommendedModel 匹配该工具对应的 Prompt。
渲染方式：紧凑型 Prompt 卡片（名称 + 场景 + 一键复制）。
文案引导：「用 ChatGPT 试试这些 Prompt，30 秒出结果」
```

#### 工具页 → 学习路径

```
规则：如果该工具出现在某条学习路径中，在底部展示路径入口。
筛选逻辑：遍历 LEARNING_PATHS，检查 days[].relatedTools 是否包含该工具 ID。
渲染方式：路径卡片（路径名 + 步数 + 适合人群）。
文案引导：「想系统学习？加入这条路径，5 步掌握完整工作流」
```

#### 工具页 ↔ 模型页

```
规则：工具页中出现「底层模型」时，链接到对应模型页。
       模型页中 relatedTool 字段链接回工具页。
实现：ChatGPT 工具页提到 GPT-5 时，可点击跳转到 GPT-5 模型页。
       GPT-5 模型页顶部标注「通过 ChatGPT 使用该模型 →」。
```

#### 实战指南 → 工具页

```
规则：教程中提到的工具名自动关联工具页。
实现：教程详情页右侧或底部展示「本教程用到的工具」。
已有基础：当前教程数据中没有 relatedTools 字段，
          但可以通过 tutorialId 反查 tools.json 中的关联。
```

#### 模型页 → Prompt 库

```
规则：模型页底部展示适合该模型的 Prompt。
筛选逻辑：按 Prompt 的 recommendedModel 字段匹配。
文案引导：「用 Claude 的这些 Prompt 效果最好」
```

### 联动实现的最小改动

| 改动 | 具体内容 | 代码量 |
|------|---------|--------|
| tools.json 扩展 | 每个工具新增 8-10 个字段 | 数据填写 |
| models.json 扩展 | 每个模型新增 8-10 个字段 | 数据填写 |
| renderToolDetailCard 升级 | 从卡片变为展开式详情，渲染新字段 | 约 100 行 |
| renderModelCard 升级 | 从卡片变为展开式详情，渲染新字段 | 约 80 行 |
| 关联内容组件 | 新建 renderRelatedContent 函数，复用于工具/模型/教程 | 约 60 行 |
| **总代码改动** | | **约 250 行 + 数据填写** |

## 7. 对现有页面最小改造建议

### 改造原则

不新建页面、不改导航、不动布局。只在现有卡片基础上增加展开详情能力。

### 改造方案

#### 方案：卡片点击展开详情面板

```
当前行为：工具卡片不可点击（只有「访问官网」和「查看教程」按钮）
改造后：点击工具卡片 → 在卡片下方展开详情面板

展开面板渲染逻辑：
1. 检查该工具是否有扩展字段（verdict 等）
2. 如果有 → 渲染完整决策型详情
3. 如果没有 → 渲染现有信息 + 「更多信息即将更新」
```

这样可以做到：
- **渐进升级**：已填写扩展字段的工具展示完整决策页，未填写的保持现有展示
- **不动布局**：工具库列表页结构不变，只是卡片可展开
- **不影响现有功能**：「访问官网」和「查看教程」按钮保持不变

#### 前端最小改动清单

```javascript
// 1. renderToolDetailCard 中增加点击事件
// 点击卡片 → 展开/收起详情面板
onclick="toggleToolExpand('${tool.id}')"

// 2. 新增 renderToolExpandedPanel 函数
function renderToolExpandedPanel(tool) {
  if (!tool.verdict) return ''; // 无扩展数据则不渲染
  return `
    <div class="tool-expanded-panel">
      <!-- verdict -->
      <!-- bestFor / notFor -->
      <!-- strengths / limitations -->
      <!-- pricingAdvice -->
      <!-- vsAlternatives -->
      <!-- quickStart -->
      <!-- relatedContent -->
    </div>
  `;
}

// 3. 新增 toggleToolExpand 函数
// 管理展开/收起状态
```

#### 模型页同理

模型卡片点击后展开详情面板，渲染扩展字段。

### 数据填写优先级

**第 1 天**：填写 ChatGPT、Claude、Perplexity 三个工具页的扩展字段，上线第一批决策型工具详情。

**第 2 天**：填写 GPT-5 系列和 Claude Sonnet/Opus 模型页的扩展字段。改造 renderModelCard 支持展开。

**第 3-5 天**：按优先级矩阵依次填写其余工具和模型。

### 关键原则：渐进发布

不需要等所有工具都填完扩展字段才上线。做完一个发一个：

- ChatGPT 工具页扩展字段填完 → 立刻上线 → 该卡片可展开详情
- Claude 工具页扩展字段填完 → 立刻上线 → 该卡片可展开详情
- 其余工具保持现有卡片样式不变 → 用户无感知

这样每天都有可见的进展，不需要一次性大改。
