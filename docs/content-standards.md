# FiveSeven AI 内容标准体系 v1.0

---

## 1. 项目理解

### 当前最需解决的内容问题

FiveSeven AI 已有 70+ 教程、500+ Prompt、40+ 工具，内容规模不小。但存在三个核心问题：

1. **质量不均匀**：部分教程信息密度高、步骤清晰（如 t001 ChatGPT 入门），部分内容偏向功能罗列，缺乏判断和结论
2. **工具页过于简介**：tools.json 中每个工具只有一段 desc + problemsSolved 列表，用户无法据此做出选择——缺少对比视角、适用边界、真实评价
3. **Prompt 缺少使用场景闭环**：prompts.json 有高质量模板，但没有「什么时候该用 / 什么时候不该用 / 用完之后怎么验证效果」的指引

### 为什么先做内容标准，不先做新内容

- 没有标准就批量生产 = 用高成本制造低质量内容
- 先定义「什么是好内容」，后续每篇新内容都能自检
- 标准体系让 AI agent 可以直接参与内容审核，降低人工负担
- 避免返工：没有标准的内容，后期全部要回炉改写

### 本次输出对后续 MVP 的帮助

| MVP 阶段 | 本次标准的作用 |
|----------|---------------|
| MVP1：内容质量提升 | 直接用模板和审核清单改写现有内容 |
| MVP2：学习路径优化 | 用评分体系筛选路径中的薄弱教程 |
| MVP3：用户互动增强 | 风格规范确保互动内容语气一致 |
| MVP4：专栏扩展 | 新专栏内容直接按模板生产，无需重新定义标准 |

## 2. FiveSeven AI 内容总原则

### 高质量内容的定义

一篇高质量内容必须同时满足以下四个条件：

1. **有明确的用户问题**：用户带着什么问题来，读完能解决什么问题
2. **有可验证的产出**：读完后用户能产出一个具体结果（一份报告、一个 Prompt、一个工作流）
3. **有判断和立场**：不是罗列所有选项，而是告诉用户「在你的场景下，选这个」
4. **有进阶路径**：读完知道下一步该学什么、该做什么

### 低质量内容的典型特征

以下任何一条出现，即判定为低质量：

- **功能搬运**：把官方文档翻译了一遍，没有加入任何自己的判断
- **万能句式**：「可以用于多种场景」「适合各种需求」——等于什么都没说
- **信息平铺**：列了 10 个功能点，但不告诉用户哪个最重要、先用哪个
- **缺少例子**：讲了方法论但没有一个可复制的操作示例
- **过时信息**：引用了已退役的模型、已下线的功能，且未标注
- **无限定词**：没有标注适用范围、局限性、失败场景

### 内容必须为用户创造的价值

每篇内容发布前，必须能回答这个问题：**「用户花 X 分钟读完，具体得到了什么？」**

不接受的答案：
- 「了解了 XX 工具」→ 了解不是价值，能用才是
- 「知道了 XX 概念」→ 知道不是价值，能判断才是
- 「学到了 XX 方法」→ 学到不是价值，能执行才是

### 内容应让用户获得的 3 种结果

| 结果层级 | 定义 | 检验标准 |
|----------|------|---------|
| **理解** | 用户能用自己的话解释这个概念/工具/方法 | 内容中有「一句话总结」或「核心区别」 |
| **判断** | 用户能决定在自己的场景下该不该用、怎么选 | 内容中有对比、推荐、适用条件 |
| **行动** | 用户能立刻动手操作，产出一个结果 | 内容中有可复制的步骤、模板、示例 |

每篇内容至少覆盖其中两个层级。实战指南必须三个全覆盖。

## 3. 实战指南模板

### 页面定位

实战指南是 FiveSeven AI 的核心内容资产。它不是「功能介绍」，而是「带着用户完成一个任务」。

一篇好的实战指南 = 一个有经验的同事坐在你旁边，手把手教你完成一件事。

### 适合解决的用户问题

- 「我想用 AI 完成 XX 任务，具体怎么操作？」
- 「这个工具我下载了，但不知道怎么用到我的工作里」
- 「我看了很多介绍文章，但还是不会动手」

### 标准结构

```
H1: [动作] + [目标结果]
    示例：「用 Claude 生成一份完整的竞品分析报告」
    反例：「Claude 竞品分析功能介绍」

## 开篇：这篇指南帮你解决什么
    - 你将学会什么（1-2 句话）
    - 最终产出是什么（具体可交付物）
    - 预计耗时
    - 前置条件（需要什么工具/账号/基础知识）

## 核心概念（仅在需要时出现）
    - 用最短篇幅解释必须知道的概念
    - 不超过 3 个概念
    - 每个概念用一句话 + 一个类比解释

## 步骤 1：[动词开头的步骤标题]
    ### 做什么
    ### 为什么这样做（不超过 2 句）
    ### 具体操作
    ### 示例/截图
    ### 容易踩的坑

## 步骤 2-N：同上结构

## 完整示例
    - 一个从头到尾的完整案例
    - 包含输入和输出
    - 标注关键决策点

## 常见问题
    - 3-5 个真实会遇到的问题
    - 直接给答案，不绕弯

## 下一步
    - 推荐 1-2 篇进阶教程
    - 一个可以立刻尝试的练习任务
```

### 每个模块的写作要求

| 模块 | 写作要求 |
|------|---------|
| 标题 | 必须包含动词 + 具体产出物。字数 15-30 字 |
| 开篇 | 30 秒内让用户知道「这篇文章值不值得我花时间看」 |
| 步骤标题 | 动词开头，描述用户要做的事，不是系统的功能名 |
| 步骤正文 | 先写操作，再解释原因。每步不超过 200 字 |
| 示例 | 必须是用户可以直接复制粘贴使用的。用游戏行业真实场景 |
| 小贴士 | 只写「如果不知道这个，你会踩坑」的信息 |
| 常见问题 | 只写「真的会被问到」的问题，不要凑数 |

### 常见错误

1. 把「功能介绍」伪装成「实战指南」——全篇在讲工具有什么功能，没有教用户做什么
2. 步骤之间缺少逻辑连接——步骤 3 突然需要步骤 1 没提到的前置操作
3. 示例太泛——用「某公司」「某产品」代替具体场景
4. 把所有注意事项堆到最后——应该在对应步骤旁边提醒
5. 步骤颗粒度不一致——步骤 1 是「打开网站」，步骤 2 是「完成整个数据分析」

### 质量加分点

- 标注了「这步大约需要 X 分钟」
- 提供了「如果你时间紧张，只做步骤 1/3/5 也能获得 80% 的效果」的快速路径
- 在关键步骤放了 before/after 对比
- 提供了可下载的模板文件
- 标注了该方法的局限性和不适用场景

### 不允许出现的问题

- 开篇超过 100 字还没进入正题
- 使用「简单来说」「总的来说」「值得一提的是」等废话连接词
- 没有任何可复制的示例
- 步骤数超过 10 步且没有分阶段
- 引用过时的模型版本/功能/价格且未标注时间

## 4. 工具页模板

### 页面定位

工具页不是产品广告，而是**选型参考**。用户来这里是为了决定「这个工具适不适合我，值不值得花时间学」。

### 适合解决的用户问题

- 「这个工具跟 XX 有什么区别？我该选哪个？」
- 「这个工具免费版够用吗？」
- 「我的岗位适合用这个工具吗？」

### 标准结构

```
H1: [工具名] — [一句话定位]
    示例：「Midjourney — 目前出图质量最高的 AI 图像生成工具」
    反例：「Midjourney 介绍」

## 一句话判断
    - 用一句话告诉用户：这个工具最适合做什么，最不适合做什么

## 核心能力
    - 列出 3-5 个核心能力
    - 每个能力配一句「在游戏行业的典型用法」
    - 标注能力等级：强项 / 够用 / 弱项

## 适用场景 × 岗位
    - 用表格展示：哪些岗位 × 哪些场景最适合用这个工具
    - 标注推荐度：首选 / 备选 / 不推荐

## 价格与性价比判断
    - 列出套餐
    - 给出明确建议：「大多数用户用 XX 套餐就够了」
    - 标注免费版的真实可用程度

## 和同类工具的核心区别
    - 最多对比 2-3 个竞品
    - 只对比用户最关心的 3 个维度
    - 给出结论：「如果你需要 XX，选它；如果你需要 YY，选另一个」

## 快速上手
    - 3 步之内让用户跑通一个最小用例
    - 链接到对应的实战指南（如果有）

## 局限性
    - 诚实列出这个工具做不好的事
    - 推荐替代方案
```

### 每个模块的写作要求

| 模块 | 写作要求 |
|------|---------|
| 一句话判断 | 不超过 40 字。必须包含「适合」和「不适合」 |
| 核心能力 | 不超过 5 个。每个能力必须关联一个游戏行业场景 |
| 适用场景 | 必须用表格。不接受纯文字描述 |
| 价格判断 | 必须给出明确推荐，不能只列价格让用户自己选 |
| 竞品对比 | 必须有结论。不接受「各有优劣，看个人需求」 |
| 局限性 | 至少列 2 条真实局限。不列局限的工具页不允许发布 |

### 常见错误

1. 把官方介绍翻译了一遍当作工具页
2. 只写优点不写缺点——用户会失去信任
3. 价格信息过时但没有标注更新日期
4. 「适合所有人」——等于没有推荐
5. 和竞品对比时不敢下结论

### 质量加分点

- 标注了「更新于 YYYY-MM-DD」
- 包含了一个 30 秒快速体验路径
- 提供了该工具在不同岗位的具体使用频率参考
- 写了「如果你已经在用 XX，是否值得切换到这个工具」

### 不允许出现的问题

- 没有价格信息或价格信息过时
- 没有任何竞品对比
- 描述中使用「强大的」「领先的」「优秀的」等无信息量形容词
- 适用岗位写了 "all" 但没有解释不同岗位的不同用法
- 缺少局限性说明

## 5. Prompt 页模板

### 页面定位

Prompt 页是一个**即拿即用的工具箱**。用户来这里不是学习 Prompt 理论，而是找到一个能直接复制粘贴、稍作修改就能用的模板。

### 适合解决的用户问题

- 「我要做 XX 任务，有没有现成的 Prompt 可以用？」
- 「我写的 Prompt 效果不好，有没有更好的写法？」
- 「这个 Prompt 应该用什么模型跑效果最好？」

### 标准结构

```
H1: [任务名] Prompt 模板

## 适用场景
    - 这个 Prompt 解决什么问题（1 句话）
    - 适用岗位标签
    - 推荐模型

## Prompt 模板
    - 完整的可复制 Prompt
    - 变量用 {花括号} 标注
    - 每个变量旁边注释说明填什么

## 变量说明
    - 每个 {变量} 的填写指南
    - 好的填写示例 vs 差的填写示例

## 完整使用示例
    - 一个完整的填入变量后的 Prompt
    - 对应的 AI 输出结果摘要（标注「实测输出」）
    - 标注使用的模型和参数

## 效果优化技巧
    - 2-3 条调优建议
    - 「如果输出太长/太短/不够具体，试试加上 XX」

## 不适用的情况
    - 这个 Prompt 处理不了什么场景
    - 遇到这些情况应该用哪个替代 Prompt

## 相关 Prompt
    - 推荐 2-3 个相关模板
```

### 每个模块的写作要求

| 模块 | 写作要求 |
|------|---------|
| 场景描述 | 一句话，不超过 30 字。用「当你需要 XX 时」的句式 |
| Prompt 模板 | 必须结构化（角色 + 任务 + 要求 + 输出格式）。变量不超过 6 个 |
| 变量说明 | 每个变量必须有好/差示例对比 |
| 使用示例 | 必须是真实跑过的，不能编造 AI 输出 |
| 优化技巧 | 只写「真的能改善输出质量」的技巧，不写泛泛的建议 |
| 不适用场景 | 至少 1 条。没有万能的 Prompt |

### 常见错误

1. Prompt 模板中变量太多（超过 6 个），用户填不完就放弃了
2. 没有使用示例——用户不知道填完变量后 Prompt 长什么样
3. 推荐模型不标注——不同模型对同一 Prompt 效果差异很大
4. notes 写得太笼统——「可根据需要调整」等于没写
5. 缺少输出格式约束——Prompt 里没有指定输出格式，导致输出不稳定

### 质量加分点

- 标注了该 Prompt 在不同模型上的效果差异
- 提供了「简化版」（适合快速使用）和「完整版」（适合精细场景）两个版本
- 附带了输出质量自检清单
- 标注了 Prompt 的适用 token 范围（避免超出模型限制）

### 不允许出现的问题

- Prompt 模板中没有角色设定
- 变量没有用 {花括号} 标注或者标注不一致
- 没有完整使用示例
- 推荐模型字段为空
- Prompt 超过 500 字但没有提供简化版

## 6. 发布前审核清单

### 第一层：基础合格项（全部通过才能发布）

- [ ] 标题包含具体的任务或产出物，不是泛泛的概念词
- [ ] 有明确的目标用户标注（岗位 / 经验水平）
- [ ] 有至少一个可复制、可执行的示例
- [ ] 所有提到的工具/模型/功能在发布日期可用且信息准确
- [ ] 没有「强大的」「优秀的」「丰富的」等无信息量形容词
- [ ] 每个步骤/模块都有实际内容，没有「待补充」或占位文字
- [ ] 字段完整：id、title、desc、difficulty、targetJobs 等必填字段均已填写
- [ ] 内部链接（nextGuides、tutorialId、columnId）指向真实存在的内容
- [ ] 没有引用已退役的模型版本或已下线的功能

### 第二层：高质量项（满足 5/8 以上为优质内容）

- [ ] 开篇 50 字内说清楚用户能得到什么
- [ ] 包含「适用 / 不适用」的边界说明
- [ ] 有竞品对比或替代方案推荐
- [ ] 示例使用了游戏行业的真实场景（不是「某公司」「某产品」）
- [ ] 每个步骤的文字量均匀，没有头重脚轻
- [ ] 包含常见错误或踩坑提醒
- [ ] 有明确的「下一步」引导
- [ ] 关键信息标注了时效性（「截至 2026 年 3 月」）

### 第三层：一票否决项（出现任何一条，直接打回）

- [ ] 内容主体是从官方文档直接翻译/复制的，没有加入自有判断
- [ ] 核心信息错误（价格错误、功能描述错误、模型版本错误）
- [ ] 没有任何可操作的示例或模板
- [ ] 标题和内容严重不匹配（标题承诺的内容正文没有兑现）
- [ ] 内容中出现未经验证的数据或结论且未标注来源

## 7. 100 分制评分模型

### 评分维度与权重

| 维度 | 权重 | 满分 | 说明 |
|------|------|------|------|
| 实用性 | 20% | 20 分 | 用户读完能做到什么 |
| 信息密度 | 15% | 15 分 | 单位篇幅传递的有效信息量 |
| 结构完整度 | 15% | 15 分 | 是否覆盖了模板要求的所有模块 |
| 判断与结论 | 15% | 15 分 | 是否有明确的推荐、对比、立场 |
| 用户适配度 | 10% | 10 分 | 是否准确匹配了目标用户的认知水平和需求 |
| 可执行性 | 15% | 15 分 | 示例和步骤是否能直接执行 |
| 成长引导感 | 10% | 10 分 | 是否让用户感觉在进步、知道下一步 |

### 各维度评分标准

#### 实用性（20 分）

| 分数 | 标准 |
|------|------|
| 18-20 | 用户读完可以直接产出一个可交付物（报告/方案/Prompt/工作流） |
| 14-17 | 用户读完能完成一个具体操作，但需要一些额外补充 |
| 10-13 | 用户读完知道方向，但还需要查找其他资料才能动手 |
| 5-9 | 用户读完只是「了解了」，无法动手 |
| 0-4 | 纯概念介绍，对实际工作没有帮助 |

#### 信息密度（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | 每段都有新信息，无冗余。删掉任何一段都会丢失关键信息 |
| 10-12 | 大部分段落有实质内容，少量过渡文字可删 |
| 7-9 | 有效信息占比约 60%，有明显的水分段落 |
| 4-6 | 大量重复表述、无意义过渡、常识性内容 |
| 0-3 | 主体是套话、废话，有效信息不足 30% |

#### 结构完整度（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | 完全符合对应模板结构，每个模块都到位 |
| 10-12 | 主体结构完整，1-2 个次要模块缺失 |
| 7-9 | 有基本结构但缺少重要模块（如缺少示例或缺少局限性说明） |
| 4-6 | 结构混乱，信息组织无逻辑 |
| 0-3 | 纯平铺文字，没有结构化组织 |

#### 判断与结论（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | 每个关键选择点都有明确推荐和理由，有对比分析 |
| 10-12 | 有推荐但部分推荐缺少理由支撑 |
| 7-9 | 有零星的判断，但关键选择点上回避了结论 |
| 4-6 | 「各有优劣，看需求」式的伪结论 |
| 0-3 | 纯罗列，完全没有判断 |

#### 用户适配度（10 分）

| 分数 | 标准 |
|------|------|
| 9-10 | 目标用户明确，语言和深度完全匹配用户认知水平 |
| 7-8 | 目标用户明确，但部分内容偏深或偏浅 |
| 5-6 | 目标用户模糊，内容深浅不一 |
| 3-4 | 没有考虑目标用户，写法像给专家或像给完全外行 |
| 0-2 | 用户定位完全错误 |

#### 可执行性（15 分）

| 分数 | 标准 |
|------|------|
| 13-15 | 所有示例可直接复制使用，步骤可直接跟做，产出可直接交付 |
| 10-12 | 大部分步骤可直接执行，少数需要用户自行补充 |
| 7-9 | 有示例但不够具体，需要用户做较多改编 |
| 4-6 | 只有方向指引，没有可直接执行的步骤 |
| 0-3 | 纯理论，没有任何操作指引 |

#### 成长引导感（10 分）

| 分数 | 标准 |
|------|------|
| 9-10 | 用户明确知道自己学到了什么、处于什么水平、下一步该学什么 |
| 7-8 | 有下一步推荐，但学习进度感不够清晰 |
| 5-6 | 内容结束就结束了，没有引导 |
| 3-4 | 有引导但指向不明确 |
| 0-2 | 用户读完感觉「然后呢？」 |

### 评分结果分级

| 分数段 | 等级 | 处理方式 |
|--------|------|---------|
| 85-100 | A 级（标杆内容） | 直接发布，可作为示范 |
| 70-84 | B 级（合格内容） | 发布，标注后续优化项 |
| 55-69 | C 级（需改进） | 改写后重新评分，达到 B 级才能发布 |
| 40-54 | D 级（不合格） | 重大改写或重写 |
| 0-39 | F 级（拒绝发布） | 不符合 FiveSeven AI 内容标准，需从头制作 |

## 8. 统一内容风格规范

### 语气

**基调：专业同事的对话语气**

- 不是教科书，不是公众号爆文，不是产品广告
- 像一个干了 5 年的同事在茶水间教你用新工具
- 自信但不傲慢，直接但不粗暴

**允许的语气**：
- 「这里有个坑，很多人第一次都会踩」
- 「说实话，这个功能对大多数人来说用不上」
- 「如果你只记住一件事，记住这个」

**不允许的语气**：
- 「在当今 AI 飞速发展的时代……」（公众号开头式）
- 「这款工具功能十分强大……」（产品介绍式）
- 「相信通过本文的学习，您一定能够……」（教科书式）

### 写法

**原则：先结论后解释，先操作后原理**

1. 每个段落的第一句话必须是这段的核心信息
2. 如果一句话能说清楚，不要用三句话
3. 用「所以」而不是用「因此」，用「但」而不是用「然而」
4. 数字用阿拉伯数字，不用中文数字
5. 英文工具名不翻译（ChatGPT 就是 ChatGPT，不写「聊天机器人」）

### 结构习惯

- 每篇内容的前 3 行必须回答「这篇文章帮你解决什么问题」
- 每个 H2 下面的第一段必须说清楚这个章节的要点
- 列表项不超过 7 条。超过 7 条时分组
- 表格优先于列表，列表优先于段落
- 步骤类内容必须编号，不用无序列表

### 长短建议

| 内容类型 | 建议长度 | 最短 | 最长 |
|----------|---------|------|------|
| 实战指南 | 2000-4000 字 | 1500 字 | 5000 字 |
| 工具页 | 800-1500 字 | 600 字 | 2000 字 |
| Prompt 页 | 500-1000 字（不含 Prompt 本体） | 300 字 | 1500 字 |
| 新闻摘要 | 100-200 字 | 80 字 | 300 字 |

### 如何避免空话

**替换规则表**：

| 空话 | 替换为 |
|------|--------|
| 功能强大 | 具体说能做什么：「支持一次处理 200K token 的长文档」 |
| 效果显著 | 给数字：「实测输出质量比 v3 提升约 40%」 |
| 广泛应用 | 列具体场景：「常用于活动策划、数据周报、竞品分析」 |
| 极大提升效率 | 给对比：「原来需要 2 小时的报告，用这个模板 20 分钟完成」 |
| 可以用于多种场景 | 删掉这句话，直接列出 3 个最典型的场景 |
| 建议根据实际情况调整 | 删掉这句话，给出 2 个调整方向的具体建议 |
| 适合各类用户 | 指定用户：「适合运营和品牌岗位，策划岗可参考但需调整」 |

### 如何体现「真干货」

1. **有判断**：「这 3 个工具里，大多数情况下选 Claude 就够了，除非你需要实时搜索」
2. **有对比**：用表格对比关键差异，最后给出结论
3. **有边界**：「这个方法适用于日活 10 万以下的游戏，大 DAU 产品需要调整参数」
4. **有失败案例**：「如果你的输入数据质量差，这个 Prompt 的输出也会不准」
5. **有个人经验**：「我们团队测试下来，参数调到 0.7 效果最稳定」

### 如何让用户感觉自己在进步

1. **标注难度等级**：每篇内容明确标注 beginner / intermediate / advanced / expert
2. **可视化进度**：在专栏和学习路径中展示「你在第 X 阶段，共 Y 阶段」
3. **明确前后关系**：「读这篇之前需要先完成 XX」「读完这篇你可以进入 XX」
4. **产出驱动**：每篇内容结束时，用户应该已经产出了一个东西——一份报告、一个 Prompt、一个工作流
5. **难度递进**：同一个主题的内容，后篇应该比前篇要求更高的操作、更复杂的判断
6. **回顾锚点**：在进阶内容中偶尔回顾「在入门篇我们学过 XX，现在你可以用它来做更复杂的 YY」

## 9. 给内容团队的执行建议

1. **先改存量，再产增量**：用评分模型给现有 70 篇教程打分，把 C 级以下的列入改写队列。新内容暂缓，直到存量内容平均分达到 B 级。

2. **工具页是当前最大短板**：tools.json 中的工具页目前只有一段描述，缺乏判断和对比。建议优先按新模板重写 5 个核心工具页（ChatGPT、Claude、Midjourney、Perplexity、Cursor），作为模板示范。

3. **每篇内容生产前先填「内容定位卡」**：在开始写作前，先用一段话回答——目标用户是谁、解决什么问题、读完的具体产出是什么。如果填不出来，这篇内容不应该开始写。

4. **用审核清单做发布门禁**：所有内容发布前，必须通过第一层全部检查项 + 第三层零命中。AI agent 可以自动执行这个审核流程。

5. **示例必须用游戏行业场景**：FiveSeven AI 的差异化就是垂直于游戏行业。所有示例必须使用真实的游戏行业场景（活动策划、投放素材、竞品分析等），不能用通用场景。

6. **每月更新一次时效性内容**：工具价格、模型版本、功能变更这些信息变化快。建立月度巡检机制，更新所有含时效性信息的页面，并在页面标注「最后更新于 YYYY-MM-DD」。

7. **Prompt 模板必须实测后再发布**：prompts.json 中的每个 Prompt 都应该在推荐模型上实际跑过，记录实测输出摘要。没有实测的 Prompt 不发布。

8. **给内容加「信息密度检查」**：写完后，逐段问自己「如果删掉这段，用户会缺少什么关键信息？」如果答案是「不会」，删掉。

9. **建立内容之间的交叉引用**：每篇教程关联对应的工具页和 Prompt 模板，每个 Prompt 反向链接到使用它的教程。让内容形成网络，而不是孤岛。

10. **每季度用评分模型做全量审计**：对所有内容重新打分，淘汰持续低于 55 分的内容，将 70-84 分的内容列入优化队列。保持内容库的整体质量水位。
