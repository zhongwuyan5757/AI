# FiveSeven AI batch-1 小范围上线与内部测试方案

> 产出日期：2026-03-12
> 目标：把 batch-1 已完成的内容以最小风险上线，发给内部同事做第一轮真实反馈验证

## 1. 小范围上线策略

### 结论：分 2 批上线，间隔 1 天

不建议一次性全上，也不建议拆得太细。2 批是最合适的节奏。

#### 第 1 批（Day 1）：3 个工具页 + 3 个 Prompt 页

| 上线内容 | 改动范围 |
|---------|---------|
| ChatGPT 工具页（样板 B） | tools.json 扩展字段 + renderToolDetailCard 改造 |
| Claude 工具页 | 同上 |
| Midjourney 工具页 | 同上 |
| p001 活动策划 Prompt 页（样板 C） | prompts.json 扩展字段 + renderPromptCard 改造 |
| p004 PRD Prompt 页 | 同上 |
| p009 数据周报 Prompt 页 | 同上 |

**为什么先上这 6 个**：

| 角度 | 理由 |
|------|------|
| 风险控制 | 工具页和 Prompt 页是「卡片展开」模式——新字段渲染失败最多是展开区域为空，不影响首页和列表页。已有内容不受影响 |
| 用户感知 | 这 6 个页面覆盖了用户最常访问的两个入口（工具库和 Prompt 库）。升级前后的体验差异最直观——从一段描述变成完整决策页/任务页 |
| 验证效率 | 一次上 6 个页面，测试者可以在工具页和 Prompt 页之间跳转，验证「站内联动」是否成立。只上 1-2 个页面联动关系无法验证 |

#### 第 2 批（Day 2-3）：2 个教程改写 + 路径 + 专题入口

| 上线内容 | 改动范围 |
|---------|---------|
| t001 ChatGPT 入门改写 | tutorials.json 中 t001 的 content 字段替换 |
| t009 活动策划改写 | tutorials.json 中 t009 的 content 字段替换 |
| 路径 A：运营 AI 上手路径 | 新增路径数据（复用 LEARNING_PATHS 结构）+ 路径渲染 |
| 专题入口页：运营 AI 方案全书 | 新增路由 + 专题页渲染组件 |

**为什么第 2 批**：

| 角度 | 理由 |
|------|------|
| 风险控制 | 教程改写是「替换现有内容」而非「新增字段」，改错了会直接影响用户正在看的教程。需要第 1 批上线确认渲染无误后再动教程 |
| 用户感知 | 路径和专题入口页是新增页面，需要第 1 批的 6 个页面都就位后才有内容可挂接。先上第 1 批，路径和专题的「推荐阅读」链接才不会指向空页面 |
| 验证效率 | 第 1 批上线后立即可以开始收工具页和 Prompt 页反馈。第 2 批上线后追加路径和专题的反馈，不需要等全部就绪才开始测试 |

#### 上线方式建议

```
Day 1 上午：在 tools.json 中给 chatgpt/claude/midjourney 添加扩展字段
Day 1 上午：在 prompts.json 中给 p001/p004/p009 添加扩展字段
Day 1 上午：改造 renderToolDetailCard 和 renderPromptCard 支持新字段
Day 1 下午：自测通过 → 部署 → 发内部测试通知

Day 2 上午：替换 tutorials.json 中 t001 和 t009 的 content
Day 2 上午：新增路径数据到 LEARNING_PATHS 或 data/paths.json
Day 2 下午：新增专题入口页组件和路由
Day 3 上午：自测通过 → 部署 → 发第 2 轮测试通知
```

## 2. 上线前最小人工检查清单

### 2.1 必须上线前检查（5 项）

#### ① 工具定价信息是否准确

| 字段 | 内容 |
|------|------|
| 检查什么 | ChatGPT、Claude、Midjourney 三个工具页中的价格和套餐名称 |
| 为什么重要 | 价格错误会直接摧毁内容信任——用户按你的建议付费后发现价格不对，会认为整个网站不可信 |
| 怎么快速检查 | 打开 chat.openai.com、claude.ai、midjourney.com 三个官网的定价页面，逐项对照。3 分钟 |
| 不检查的风险 | 用户截图传播「FiveSeven AI 价格信息有误」，品牌损伤 |

#### ② 所有 Prompt 模板是否可正确复制

| 字段 | 内容 |
|------|------|
| 检查什么 | p001/p004/p009 的「一键复制」按钮是否正常工作，复制后的文本是否完整且无乱码 |
| 为什么重要 | Prompt 页的核心价值就是「复制即用」。复制按钮坏了 = 页面最核心功能失效 |
| 怎么快速检查 | 在页面上点「一键复制」，粘贴到记事本检查。每个 Prompt 30 秒，共 1.5 分钟 |
| 不检查的风险 | 用户第一印象就是「这个 Prompt 还得手动选中复制」，直接离开 |

#### ③ 站内跳转链接是否指向真实存在的页面

| 字段 | 内容 |
|------|------|
| 检查什么 | 每个页面底部的「相关内容」链接是否都能点开且不是 404 |
| 为什么重要 | 「下一步」引导是这次升级的核心卖点。链接 404 = 引导断裂 = 用户流失 |
| 怎么快速检查 | 从 ChatGPT 工具页开始，沿着推荐链接点 5 跳，看是否每步都能到达。2 分钟 |
| 不检查的风险 | 内部测试者第一条反馈就是「链接打不开」，浪费整轮测试机会 |

#### ④ 新增字段在移动端是否正常显示

| 字段 | 内容 |
|------|------|
| 检查什么 | 工具页的「竞品对比表格」和 Prompt 页的「变量填写指南表格」在手机上是否溢出或被截断 |
| 为什么重要 | 内部同事大概率在手机上看你发的链接。表格溢出是移动端最常见的显示问题 |
| 怎么快速检查 | 用 Chrome DevTools 切换到 iPhone/Android 视图，检查 3 个页面的表格区域。3 分钟 |
| 不检查的风险 | 手机上看到一堆横向滚动的表格，「干货感」大打折扣 |

#### ⑤ 非升级页面是否正常显示（回归测试）

| 字段 | 内容 |
|------|------|
| 检查什么 | 没有升级的工具页（如 Perplexity、Cursor）和 Prompt 页（如 p002、p003）是否还正常显示 |
| 为什么重要 | 改造 renderToolDetailCard 时如果没有做好「新字段不存在时的降级处理」，会导致未升级页面白屏或报错 |
| 怎么快速检查 | 点开 2 个未升级的工具页 + 2 个未升级的 Prompt 页，确认无白屏。1 分钟 |
| 不检查的风险 | 35 个工具页中只升级了 3 个，如果其余 32 个全挂了 = 严重线上事故 |

### 2.2 可上线后 1-3 天内补修（4 项）

| # | 检查什么 | 为什么可以晚修 | 什么时候修 |
|---|---------|-------------|-----------|
| ① | 工具页 lastUpdated 日期是否准确 | 不影响功能，只影响信息时效感知 | 上线后第 1 天 |
| ② | Prompt 页 exampleFull 中的输出摘要是否都经过实测验证 | 不影响使用，但会影响信任度。可以上线后安排实测补充 | 上线后第 2 天 |
| ③ | 竞品对比中「选这个/选那个」的措辞是否有偏颇或遗漏 | 不影响功能，可以根据内部反馈调整措辞 | 收到反馈后调整 |
| ④ | 教程改写中的 Prompt 示例是否在最新模型版本上跑过 | 不影响结构和阅读体验，但如果示例过时会影响「实测」可信度 | 上线后第 3 天前完成 |

## 3. 内部测试对象建议

### 推荐 5-8 人，分 3 类

| 类型 | 人数 | 具体是谁 | 为什么有价值 |
|------|------|---------|-------------|
| **运营/策划岗同事** | 2-3 人 | 日常做活动策划、写 PRD、做数据周报的人 | 他们是目标用户。能判断 Prompt 是否真的好用、变量指南是否有帮助、工具对比是否符合实际使用体验 |
| **已有 AI 使用经验的同事** | 1-2 人 | 平时就在用 ChatGPT/Claude 工作的人 | 能判断工具页的竞品对比是否准确、定价建议是否合理、局限性描述是否到位。他们会发现「说得不对」的地方 |
| **AI 零基础的同事** | 2-3 人 | 听说过 AI 但没怎么用、或者试过但放弃了的人 | 能验证 t001 入门教程和路径 A 的「零基础上手」体验是否真的成立。如果他们看完还是不知道从哪开始，说明入门引导有问题 |

### 为什么不要超过 8 人

| 理由 | 解释 |
|------|------|
| 反馈收集成本 | 超过 8 人，你需要花大量时间整理和去重反馈，效率急降 |
| 反馈质量 | 人多了反馈变成「挺好的」「不错」这种无用信息。少而精的测试者能给出具体问题 |
| 迭代速度 | 5-8 人的反馈 1 天内能看完、2 天内能改完。20 人的反馈你一周都处理不完 |
| 心理压力 | 人太多会让测试者觉得「随便看看」，人少反而有「被认真邀请」的参与感 |

## 4. 内部测试问题清单

### 4.1 最推荐的 5 个核心问题

每个问题对应一个核心验证目标。

#### 问题 1：路径感

> 你打开 ChatGPT 工具页后，是否清楚「下一步该看什么」？你实际点了「下一步」的链接吗？如果没点，是什么原因？

**为什么问**：这次升级的核心改进之一是「每个页面都有下一步引导」。如果用户看完一个页面还是不知道去哪，说明联动设计失败。

#### 问题 2：决策感

> 看完 ChatGPT 工具页后，你是否更清楚「ChatGPT 和 Claude 我该选哪个」了？如果还是不确定，缺了什么信息？

**为什么问**：工具页从「介绍型」升级为「决策型」，核心假设就是「对比信息能帮用户做选择」。如果看完还是不确定，说明对比维度选错了或结论不够明确。

#### 问题 3：可用性

> 打开 p001 活动策划 Prompt 页，你是否觉得自己「敢用」这个 Prompt 了？「变量填写指南」对你有帮助吗？如果让你现在就用它完成一个任务，你觉得还缺什么？

**为什么问**：Prompt 页升级的核心假设是「加了 inputGuide 和 commonFailures，用户不再只是复制走人，而是会用、敢用」。这个问题直接验证假设。

#### 问题 4：干货感

> 和网站上其他没有升级的页面相比（比如随便点一个没升级的工具或 Prompt），你觉得升级后的页面「更有干货」吗？最大的差别是什么？

**为什么问**：用户不会看你的内部标准和评分模型，他们的判断就是「有没有干货」。这个问题直接测量升级前后的感知差异。不需要用户精确描述好在哪——他们能感知到差异就说明方向对了。

#### 问题 5：最该修的一个问题

> 如果你只能提一个改进建议，你觉得这些页面最该改的是什么？

**为什么问**：限定「只能提一个」迫使测试者说出他认为最重要的问题，而不是列一堆无关痛痒的小问题。5 个人的「最该改的一个问题」合在一起，就是你的优先修复清单。

### 4.2 可补充追问的 3 个问题

用在测试者回复了核心问题之后，针对性追问。

#### 追问 A：入门体验（问 AI 零基础的测试者）

> 你看了 t001 入门教程后，是否知道怎么开始？如果让你现在就按教程做一遍，卡在哪里？

#### 追问 B：信任度（问有 AI 经验的测试者）

> 工具页里的竞品对比和定价建议，有没有哪条让你觉得「这个说得不对」或「这个太偏向某个产品了」？

#### 追问 C：路径完整度（问走完路径 A 的测试者）

> 你走完路径 A 的 5 步后，觉得自己是否「会用 AI 做日常工作了」？哪一步跳跃太大或者内容不够？

## 6. 上线后 3 天内最该观察的信号

不需要埋点工具、不需要数据平台。以下 5 个信号用肉眼观察 + 测试者反馈即可获取。

### 信号 1：用户是否点了「下一步」

| 字段 | 内容 |
|------|------|
| 看什么 | 测试者是否从工具页跳到了 Prompt 页或教程，或者从 Prompt 页跳到了教程 |
| 为什么重要 | 「站内联动」是这次升级最核心的结构改进。用户不跳 = 联动设计失败 |
| 方向对了的信号 | 测试者反馈「我从工具页点到了 Prompt 页，觉得很自然」或者「我顺着推荐看了 3 个页面」 |
| 要立刻修的信号 | 测试者反馈「没注意到下面还有推荐链接」或者「看到了但不想点，因为不知道点进去是什么」 |
| 怎么修 | 调整推荐区域的视觉层级（加大字号/换颜色/加图标），或改跳转文案让用户知道点进去能得到什么 |

### 信号 2：Prompt 页有没有人真的「用了」

| 字段 | 内容 |
|------|------|
| 看什么 | 测试者是否真的复制了 Prompt 去 AI 工具里跑了一次 |
| 为什么重要 | Prompt 页的终极价值不是「看」而是「用」。复制了但不敢用 = inputGuide 没解决用户顾虑 |
| 方向对了的信号 | 测试者说「我照着填了一遍，跑出来的结果还行」或者「变量指南帮我知道了怎么填」 |
| 要立刻修的信号 | 测试者说「看了指南还是不确定怎么填」或者「Prompt 太长了我不想看」 |
| 怎么修 | 如果是不确定怎么填→补充更多好/差示例。如果是太长→把新手版放在更显眼的位置 |

### 信号 3：工具对比是否帮助了决策

| 字段 | 内容 |
|------|------|
| 看什么 | 测试者看完工具页后是否说出了类似「我知道该选哪个了」的话 |
| 为什么重要 | 工具页从「介绍」升级为「决策」的核心假设。用户看完还是说「不确定」= 对比维度选错了 |
| 方向对了的信号 | 「原来写长方案用 Claude 更好，我之前一直用 ChatGPT」或「定价建议很有用，我决定先用免费版试试」 |
| 要立刻修的信号 | 「对比看完了但还是不知道选哪个」或「感觉有点偏向 XX 工具」 |
| 怎么修 | 如果不知道选→结论句加粗加大、放在对比表格上方。如果有偏向感→平衡措辞、补充另一方的优势场景 |

### 信号 4：干货感的前后差异

| 字段 | 内容 |
|------|------|
| 看什么 | 测试者是否在没有引导的情况下主动对比了升级页面和未升级页面 |
| 为什么重要 | 如果用户自发感受到差异，说明升级方向正确且效果明显。如果需要你解释「你注意到了吗」才能感知差异，说明改动不够大 |
| 方向对了的信号 | 「这个页面比其他的详细多了」「这才像个正经推荐」「其他页面什么时候也改成这样」 |
| 要立刻修的信号 | 「我没觉得有什么不同」或「信息太多了看不完」 |
| 怎么修 | 如果没感觉不同→视觉层级不够，新增内容区域需要更强的设计区分。如果信息太多→折叠次要信息，默认只展示 verdict + 核心能力 + 快速上手 |

### 信号 5：路径有没有人走到第 2 步

| 字段 | 内容 |
|------|------|
| 看什么 | 进入路径 A 的测试者是否至少完成了前 2 步 |
| 为什么重要 | 路径的价值在于「系统化学习」。如果用户看了第 1 步就走了 = 路径结构或步骤描述没有制造「想继续」的动力 |
| 方向对了的信号 | 「我按路径走了 3 步，感觉确实在进步」 |
| 要立刻修的信号 | 「看到路径有 5 步就不想看了」或「第 1 步太简单了直接跳过了但第 2 步又太难了」 |
| 怎么修 | 如果畏难→在路径入口加「每步只需 X 分钟」的时间锚点。如果难度跳跃→在步骤间加过渡说明 |

## 7. 是否进入 batch-2 的判断标准

### 可以进入 batch-2 的条件（满足 3/4 即可）

| # | 条件 | 怎么判断 |
|---|------|---------|
| 1 | **干货感验证通过** | 至少 4/5 的测试者明确表示升级后的页面「更有干货」或「比其他页面好很多」 |
| 2 | **联动跳转成立** | 至少 3/5 的测试者实际点了「下一步」推荐链接，且没有遇到 404 或「不知道为什么推荐这个」 |
| 3 | **Prompt 页从看到用** | 至少 2/5 的测试者真的复制了 Prompt 去跑了一次，且反馈「变量指南有帮助」 |
| 4 | **没有结构性问题** | 测试者反馈中没有出现「信息太多看不完」「找不到重点」「页面加载太慢」等结构性问题 |

### 应该先修 batch-1 的条件（出现任何 1 条就先修）

| # | 信号 | 说明 | 预计修复耗时 |
|---|------|------|-------------|
| 1 | 超过 2 人反馈「信息太多了」 | 需要做信息折叠——默认展示核心区域，详细内容折叠到「展开更多」 | 2-4 小时 |
| 2 | 超过 2 人反馈「不知道下一步看什么」 | 推荐区域视觉层级不够或文案不够吸引人，需要调整 UI 和文案 | 1-2 小时 |
| 3 | 有人反馈定价或工具信息有误 | 立刻修正，信息错误是信任杀手 | 30 分钟 |
| 4 | 移动端表格显示有问题 | 需要加 CSS 适配（横向滚动或换成列表） | 1-2 小时 |
| 5 | 非升级页面受影响（白屏/报错） | 渲染函数的降级处理有 bug，紧急修复 | 1 小时 |

### 最值得优先修的 3 类问题（根据历史经验排序）

| 优先级 | 问题类型 | 为什么优先 |
|--------|---------|-----------|
| P0 | **信息错误**（价格/功能/模型版本不对） | 信任一旦丢失很难恢复。发现即修，不过夜 |
| P1 | **信息过载**（用户觉得内容太多看不完） | 这是内容升级最常见的副作用。解决方案通常是加折叠，改动小但效果立竿见影 |
| P2 | **联动断裂**（推荐链接没人点 / 文案不够吸引人） | 跳转文案是最容易迭代的部分，改几行文字就能测试效果 |

### batch-2 的启动条件总结

```
if (干货感通过 && 联动跳转成立 && 无结构性问题) {
  → 启动 batch-2：Claude 工具页已上线可对标，
    扩展 Perplexity/Cursor 工具页 + p002/p003/p005 Prompt 页
}

if (有信息错误 || 移动端问题 || 非升级页面受影响) {
  → 紧急修复，1 天内解决，然后重新发一轮测试确认
}

if (信息过载 || 联动断裂) {
  → 花 1-2 天做 UI 微调（折叠、文案优化），
    再发一轮测试验证改善效果
  → 改善确认后启动 batch-2
}
```
